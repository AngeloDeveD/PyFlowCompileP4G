Project и source

Это слой, который отвечает за структуру мода: какой скрипт патчится, создаётся ли новый container, какие сообщения и процедуры войдут в build, а также какие bits будут зарезервированы для мода.

APIКогда использоватьМинимальный пример
P4GProject.patch_script(...)Патч любого существующего .bf.P4GProject.patch_script("mod", Path("file.bf"))
P4GProject.patch_field_script(...)Патч field-скрипта с готовым target path.P4GProject.patch_field_script("mod", "f007", Path("f007.bf"))
P4GProject.new_event_script(...) / patch_event_script(...)Create/patch event script с готовым target path.P4GProject.patch_event_script("mod", "E860_074A", Path("E860_074A.bf"))
P4GProject.new_script(...)Создать новый script container. Нужен внешний вызов.P4GProject.new_script("mod", target="...")
BF / Empty / FieldBF / EventBFSource helpers для patch/create mode.SCRIPT = FieldBF("f007", Path("f007.bf"))
field_target(...) / event_target(...)Собрать стандартный target path без ручной строки.field_target("f007")
project.reserve_bit(...)Выделить bit состояния мода.done = project.reserve_bit("MOD_DONE")
project.message(...)Создать отдельный message block.msg = project.message("MOD_MSG")
project.Build(...)Собрать .flow, .msg и .bf.project.Build(Debug=True)
project.write / compile / buildНизкоуровневые этапы build pipeline.project.write("out")
project.render_flow() / render_msg()Посмотреть generated source без записи файлов.print(project.render_msg())

Hooks и Procedure

Это рабочая зона сценариста. Здесь создаются hooks, обычные процедуры и конкретные действия внутри них: вызовы native-функций, диалоги, условия, выборы, bits и переходы.

APIСмыслВажно
project.replace_hook("target")Создаёт target_hook().Оригинал вызывается только если добавить call_original().
project.soft_hook("target")Создаёт target_softhook().Подходит для before-добавок.
project.after_hook("target")Создаёт target_hookafter().Подходит для действий после оригинала.
proc.say(...)Показать короткую реплику.Сам создаёт message block и вызовы message window.
proc.ask(...)Показать выбор и вернуть переменную ответа.Первый вариант имеет индекс 0.
proc.once(bit)Выполнить блок один раз.В конце блока автоматически пишет BIT_ON(bit).
proc.call("NATIVE", ...)Прямой native-вызов FlowScript.Для неизвестных вызовов нужен safety="standard".

Expressions и conditions

Helpers этого раздела позволяют писать проверки в Python-форме, а библиотека сама переводит их в понятный FlowScript. Именно этот слой особенно полезен для режима beginner.

HelperЧто спрашивает или делаетРендерится как
bitChk(done) == 0Bit выключен?BIT_CHK(done) == 0
bitOn(done)Включить bit.BIT_ON(done);
checkTimeSpan(4, 1, 5, 6) == 1Дата внутри диапазона?CHECK_TIME_SPAN(4, 1, 5, 6) == 1
timeOfDay() == TimeOfDay.DAYСейчас нужная часть дня?GET_TIME_OF_DAY() == ...
var("choice") == 0Проверить FlowScript-переменную.choice == 0
getItem / setItem / addYenInventory/money helpers в expression-style.proc.emit(addYen(3000))
setMsgVar(...) / msg_var(...)Подготовить и вывести переменное значение в тексте.proc.set_msg_var(0, 3000)
rawExpr(...) / callExpr(...)Advanced raw expressions для standard mode.rawExpr("SOME_NATIVE()")

Messages и selections

Здесь лежат объекты, которые отвечают за текстовые ресурсы: диалоги, строки, raw-теги и варианты выбора для игрока.

APIНазначениеПример
project.message("NAME")Создать [msg NAME].msg = project.message("MOD_MSG")
message.line("text")Добавить строку в message block. Длинный текст переносится автоматически: 40 символов, максимум 3 строки.msg.line("Привет", wrap_width=35)

Diagnostics

Когда мод компилируется, но не ведёт себя как ожидалось, на сцену выходят инструменты проверки: build-логи, manifest-файлы, анализ конфликтов и doctor-диагностика.

ИнструментЧто проверяетКогда смотреть
Build(Debug=True)Подробный вывод AtlusScriptTools.Если .bf не собрался.
manifest.jsonКакие файлы, imports, target path и warnings были у build.Если Reloaded II загрузил не тот путь.
conflicts.txtПотенциальные пересечения hooks, bits и target scripts.Если несколько модов меняют один script container.
doctor / analyzerСтруктуру workspace и использование процедур.Если build есть, но hook не достигается в игре.
build_project_file(...) / load_project_from_file(...)Сборка модов из Python-файлов через tooling API.Для batch-build и launchers.
CompilerConfig / BuildResult / CompileResultStructured config/result objects build pipeline.Для автоматизации и UI.
run_doctor / analyze_project_conflictsДиагностика окружения и конфликтов проекта.Когда мод собирается, но ведёт себя странно.

P4GProject

Главный объект сборки. В нём живёт будущий мод: source, hooks, процедуры, messages, selections, bits и вызов Build().

P4GProject.patch_script(name, original_bf, *, target=None, import_path=None, safety="beginner")

Зачем нужен

Создавай один проект на один собираемый script container. Для первых модов чаще всего нужен patch-route: оригинальный .bf, hook к реальной процедуре и итоговый patched .bf.

Минимальный пример

project = P4GProject.patch_script(
    "hello_scheduler",
    Path(__file__).with_name("scheduler_01.bf"),
    target="data_e/scheduler/scheduler_01.bf",
    import_path="original_scheduler_01.bf",
    safety="beginner",
)

Что получится

Интерактивный JS-справочник заменит этот стартовый блок подробной карточкой выбранного элемента. Без JS эта карточка остаётся полезным fallback, а таблицы выше дают быстрый обзор API.

Справочник намеренно не дублирует большой урок. Если ты ещё не уверен, как игра вообще вызывает процедуры и почему hook работает, сначала возвращайся в обучение, а уже потом читай отдельные элементы API.
